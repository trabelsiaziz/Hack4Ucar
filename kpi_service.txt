2) KPI FORMULAS

===============

 

A. Academic KPIs

----------------

 

1. SUCCESS_RATE

Definition:

Percentage of students who passed in a given scope and period.

Formula:

SUCCESS_RATE = (number_of_students_passed / number_of_students_assessed) * 100

Numerator source:

- assessment_results where passed = true

Denominator source:

- assessment_results

Unit:

- percent

Typical scope:

- institution / org_unit / program / period

 

2. ATTENDANCE_RATE

Definition:

Average attendance rate for a student population in a given scope and period.

Formula:

ATTENDANCE_RATE = average(attendance_rate)

Source:

- attendance_records

Unit:

- percent

Typical scope:

- institution / org_unit / course_offering / period

 

3. DROPOUT_RATE

Definition:

Percentage of enrolled students marked as dropped out in a given period.

Formula:

DROPOUT_RATE = (number_of_dropout_students / total_enrolled_students) * 100

Numerator source:

- students or enrollments where enrollment_status = 'dropped' (depending on business rule)

Denominator source:

- enrollments

Unit:

- percent

Typical scope:

- institution / program / period

 

4. EXAM_RESULTS_AVG

Definition:

Average exam/assessment score in a given scope and period.

Formula:

EXAM_RESULTS_AVG = average(score)

Source:

- assessment_results

Unit:

- score

Typical scope:

- institution / course_offering / program / period

 

 

B. Finance KPIs

---------------

 

5. ALLOCATED_BUDGET

Definition:

Total allocated budget for a given scope and period.

Formula:

ALLOCATED_BUDGET = sum(amount where budget_type = 'allocated')

Source:

- budget_facts

Unit:

- currency

Typical scope:

- institution / org_unit / period

 

6. CONSUMED_BUDGET

Definition:

Total consumed budget for a given scope and period.

Formula:

CONSUMED_BUDGET = sum(amount where budget_type = 'consumed')

Source:

- budget_facts

Unit:

- currency

Typical scope:

- institution / org_unit / period

 

7. BUDGET_EXECUTION_RATE

Definition:

Percentage of allocated budget that has already been consumed.

Formula:

BUDGET_EXECUTION_RATE = (CONSUMED_BUDGET / ALLOCATED_BUDGET) * 100

Sources:

- budget_facts

Unit:

- percent

Typical scope:

- institution / org_unit / period

Note:

- if ALLOCATED_BUDGET = 0, return null or 0 according to business policy

 

8. COST_PER_STUDENT

Definition:

Average budget cost per enrolled student.

Formula:

COST_PER_STUDENT = CONSUMED_BUDGET / total_enrolled_students

Numerator source:

- budget_facts where budget_type = 'consumed'

Denominator source:

- enrollments

Unit:

- currency per student

Typical scope:

- institution / period

Note:

- if total_enrolled_students = 0, return null or 0 according to business policy

 

 

C. HR KPIs

----------

 

9. TEACHING_HEADCOUNT

Definition:

Number of active teachers in a given scope.

Formula:

TEACHING_HEADCOUNT = count(teachers where employment_status = 'active')

Source:

- teachers

Unit:

- count

Typical scope:

- institution / org_unit / period snapshot

 

10. ADMINISTRATIVE_HEADCOUNT

Definition:

Number of active administrative/support staff in a given scope.

Formula:

ADMINISTRATIVE_HEADCOUNT = count(staff_members where employment_status = 'active')

Source:

- staff_members

Unit:

- count

Typical scope:

- institution / org_unit / period snapshot

 

11. ABSENTEEISM_RATE

Definition:

Average absenteeism rate for staff or teacher groups.

Formula option A (using hr_facts):

ABSENTEEISM_RATE = average(value where hr_metric = 'absenteeism_rate')

Formula option B (using attendance_records):

ABSENTEEISM_RATE = (number_of_absences / total_expected_attendance_events) * 100

Primary V1 source:

- hr_facts

Optional supporting source:

- attendance_records

Unit:

- percent

Typical scope:

- institution / org_unit / period

 

12. TEACHING_LOAD

Definition:

Total teaching hours assigned to teachers.

Formula:

TEACHING_LOAD = sum(hours_assigned)

Source:

- teaching_assignments

Unit:

- hours

Typical scope:

- institution / org_unit / teacher / period

7) HOW THE BACKEND SHOULD WORK

==============================

 

Step 1 — read source/domain data

--------------------------------

Read from source fact tables:

- enrollments

- attendance_records

- assessment_results

- budget_facts

- teachers

- staff_members

- hr_facts

- teaching_assignments

 

Step 2 — normalize if needed

----------------------------

If required by the pipeline, normalize facts into:

- business_records

- business_record_values

 

Step 3 — compute KPI observations

---------------------------------

For each KPI definition and target scope:

- compute value

- apply formula version

- store result into kpi_observations

 

Step 4 — evaluate thresholds

----------------------------

Using kpi_thresholds:

- generate alerts when thresholds are exceeded

 

Step 5 — detect anomalies

-------------------------

Using rules/statistical checks:

- generate anomalies when unusual patterns are found

 

Step 6 — generate AI outputs

----------------------------

Generate:

- insights

- recommendations

- decision explanations

based on KPI observations, alerts, and anomalies

 

Step 7 — expose frontend-ready widgets

--------------------------------------

Backend transforms analytical results into:

- stat-card responses

- chart responses

- alert-list responses

- recommendation-list responses

- insight-card responses

 

 

8) NEXT DEVELOPMENT STEPS

=========================

 

Phase 1 — freeze V1 KPI scope

-----------------------------

Freeze these 12 KPIs only:

- SUCCESS_RATE

- ATTENDANCE_RATE

- DROPOUT_RATE

- EXAM_RESULTS_AVG

- ALLOCATED_BUDGET

- CONSUMED_BUDGET

- BUDGET_EXECUTION_RATE

- COST_PER_STUDENT

- TEACHING_HEADCOUNT

- ADMINISTRATIVE_HEADCOUNT

- ABSENTEEISM_RATE

- TEACHING_LOAD

 

Phase 2 — implement KPI governance seed data

--------------------------------------------

Create seed data for:

- kpi_definitions

- kpi_formula_versions

- kpi_thresholds

 

Phase 3 — implement KPI engine services

---------------------------------------

Build backend services that:

- fetch source/domain data,

- compute KPI observations,

- persist kpi_observations,

- generate alerts,

- optionally generate anomalies

 

Phase 4 — implement dashboard APIs

----------------------------------

Build backend endpoints returning:

- stat-card responses

- chart responses

- alert-list responses

- recommendation-list responses

- insight-card responses

 

Phase 5 — implement frontend widget rendering

---------------------------------------------

Connect the dynamic frontend widgets to those APIs.

 

Phase 6 — implement AI core / LangGraph

---------------------------------------

Once KPI observations and alerts are stable, build the AI core that:

- answers KPI questions,

- explains alerts,

- compares institutions,

- summarizes dashboard state,

- produces recommendations

 

Phase 7 — add explainability and traces

---------------------------------------

Ensure the AI core references:

- KPI observation IDs

- alert IDs

- anomaly IDs

- recommendation IDs

- decision explanations

 

 

9) FINAL SIMPLIFIED RULE

========================

 

- Raw/domain tables store facts.

- KPI engine computes KPI observations.

- Alerts and anomalies are monitoring outputs.

- Insights and recommendations are AI outputs.

- Frontend displays backend-prepared widget data.

- LangGraph should reason on top of KPI observations, alerts, anomalies, insights, and recommendations — not directly on raw tables whenever avoidable.
